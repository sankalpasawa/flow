## Problem Statement

Knowledge workers who already have planning habits are running a duct-taped system:
Any.do for task/time management + Google Docs for hourly blocking, intention-setting,
and reflection. No single app unifies hourly scheduling + experiential logging +
AI mindset framing. DayFlow is the consolidation play for this specific user —
not a behavior-change product, but better infrastructure for a habit they already have.

## Demand Evidence

- Founder is the target user (Persona 1: Structured Achiever)
- Currently using Any.do + Google Docs as a duct-taped workaround
- Any.do fails on all three dimensions: no reflection/logging layer, no hourly canvas,
  no mindset/intention layer — all gaps patched manually in Google Docs
- Plans to validate with specific named users this week (not yet done — see Assignment)

Signal tier: Medium. Real personal pain, real workaround, unverified external demand.

The target user's current workflow:

  experience logging, mindset framing
- **Google Docs** — hourly day plan, daily intention/theme, post-activity reflection notes

  no behavioral patterns extracted from the data they're already generating

## Approaches Considered

### Approach A: Web-first MVP
Next.js PWA + Supabase + Claude API. Ship P0 canvas + log + AI in 4-6 weeks.
- Effort: M | Risk: Low
- Pros: Fast iteration, no App Store delays, Structured Achiever is desktop-heavy
- Cons: No native push notifications — breaks the nudge system

### Approach B: Full PRD Stack
React Native + Next.js + Fastify + PostgreSQL + Redis as specified.
- Effort: XL | Risk: High
- Pros: Complete cross-platform from day 1
- Cons: 3-4 months to first user; burns runway if hook doesn't land

### Approach C: Mobile-first, Log-first (CHOSEN)
iOS first, experience log as hero feature.
- Effort: L | Risk: Medium
- Pros: Native push notifications (critical for nudge loop), log as moat from day 1,
  behavioral data collection starts immediately
- Cons: Slower to ship than web, Android/web users excluded initially

Circuit breaker: if any user exceeds 30 AI calls/day, queue remaining calls for next day.
Implementation: Circuit breaker lives in Supabase Edge Function (server-side, not client).

queue. Queue drains at midnight UTC.

## Data Model (Key Entities)

```
Activity {
  id: UUID, user_id: UUID, title: String(80),
  start_time: DateTime, duration_minutes: Int(15-240),
  category_id: FK->Category, mindset_prompt: String,
  mindset_overridden: Bool, recurrence_type: Enum,
  status: Enum(PLANNED|IN_PROGRESS|COMPLETED|SKIPPED),
  priority: Enum(HIGH|MEDIUM|LOW)
}

ExperienceLog {
  id: UUID, activity_id: FK->Activity, user_id: UUID,
  mood: Int(1-5), energy: Int(1-5), completion_pct: Int(0|50|100),
  reflection: Text, would_repeat: Enum(YES|NO|MAYBE),
  logged_at: DateTime  -- editable up to 24h after activity end
}

Category {
  id: UUID, user_id: UUID, name: String, color: HexString,
  icon: String, is_system: Bool, sort_order: Int
}

User {
  id: UUID, email: String, subscription_tier: Enum(FREE|PRO),
  settings: JSONB  -- day_start, quiet_hours, nudge_prefs
}
```

ExperienceLog definition: A log entry on a planned activity.
Timing: flexible — can be logged before, during, or after an activity (user may want to
pre-log an intention, or log mid-activity). Editing window: open until 24h after the
activity's end_time. After that window, entries are read-only.

The log prompt nudge fires based on end_time (not completion status) — fires 30min after
end_time, regardless of whether the activity was marked complete. Rationale: users often
forget to mark activities complete; end_time is a reliable signal.

**Architecture findings:**

1. **Offline-first gap (AUTO-DECIDED → add to Sprint 1):** The plan flags Supabase Realtime as risky with REST polling fallback, but doesn't specify local data persistence. A mobile app that loses data when offline is broken. **Fix:** Add local SQLite (via expo-sqlite or WatermelonDB) as the source of truth, with Supabase sync on reconnect. This is the Week 1 gate.

2. **Push notification scheduling unclear:** Who triggers the 30-min-after-end_time nudge? Options: (a) Supabase cron job polling activities, (b) client-side scheduled local notification, (c) Supabase Database Webhooks. **Fix:** Use Expo's local notification scheduling on the client. When an activity is created, schedule a local notification for end_time + 30min. No server-side cron needed for this. Server-side only for the end-of-day planning nudge.

3. **No API versioning strategy:** Supabase Edge Functions don't have built-in versioning. If the AI prompt format changes, old app versions break. **Fix:** Version Edge Function paths (/v1/ai/categorize) or use a version header.

4. **Single point of failure: Supabase.** All data, auth, realtime, and Edge Functions depend on one provider. Acceptable for MVP but should be noted. **Mitigation:** Supabase is open-source; can self-host if needed.

**Rollback posture:** TestFlight builds can be reverted by uploading a new build. DB migrations need backward compatibility (no column drops without a release cycle). Feature flags on AI features allow server-side disable.

### Section 7: Performance Review

- **Supabase query performance:** Canvas loads all activities for a day. With RLS, this is `SELECT * FROM activities WHERE user_id = $1 AND date(start_time) = $2`. Needs index on `(user_id, start_time)`.
- **Realtime subscription:** One subscription per user for activities table. Supabase handles connection pooling.
- **Claude API latency:** Haiku ~200ms, Sonnet ~1-2s. Mindset prompts are eager (at creation) so user waits. **Fix:** Show loading indicator, stream response if possible.

### Critical Design Findings (both voices agreed)

**F1 (CRITICAL): Log is the hero feature but buried behind notifications.**
The plan says "the log is the product" but the IA puts canvas first and log behind a notification tap. Users will experience this as a time-blocking app with a journaling add-on.
**Fix:** Inline log indicators on activity cards (mood/energy dots). Make tapping any activity open a detail view with prominent "Log Experience" button. Notifications are reminders, not the only path.

**F2 (CRITICAL): Empty canvas is a dead screen.**
New user after onboarding sees 24 empty hourly rows. The retention loop (nudge→log→reflect) can't start without an activity.
**Fix:** Show prominent CTA "Plan your first block" at current hour. Fire planning nudge for users with 0 activities (not just <3). Show encouraging "rest day" illustration on empty days.

**F3 (CRITICAL): Canvas has no visual specification.**
"Hourly day canvas" is a concept, not a design. Unspecified: layout type (timeline vs grid vs list), how 15-min vs 2-hour blocks look, overlap rendering, creation gesture, mindset prompt placement.
